# Plano de Otimização Sanitária RebanhoSync

**Status:** Em análise — não implementar

## 1. Resumo Executivo

A arquitetura sanitária demonstra maturidade com separação clara de camadas (official → custom → standard), mas apresenta riscos de inconsistência: `operational_complement` vs `is_operational_complement` divergente (P1), versionamento de itens não semântico (P0), chamada RPC direta após `createGesture` (P0), e checklist/regulatory confundidos com protocolos operacionais (P2). Agenda usa dedup robusto, mas o default `intervaloDias=1` pode gerar agenda indevida. Protocolos oficiais são editáveis sem proteção explícita.

## 3. Mapa dos Fluxos Atuais

### 3.1 Configuração de Protocolo

```
[UI: ProtocolosSanitarios/index.tsx]
    ↓
[Hook: useProtocolosData]
    ↓
[State tables: protocolos_sanitarios, protocolos_sanitarios_itens]
    ↓
[Schema: baseline.sql (linhas 431-473)]
```

- **Camada official**: `catalogo_protocolos_oficiais` → `catalogo_protocolos_oficiais_itens` (READ-ONLY, autenticado)
- **Camada tenant-scoped**: `protocolos_sanitarios` + `protocolos_sanitarios_itens` (RLS por fazenda)
- **Overlay**: `payload.origem="customizado_fazenda"` com `operational_complement` (boolean)
- **Versionamento**: Coluna `version` (integer, default 1) mas UI não versiona UPDATE

### 3.2 Geração/Recompute de Agenda

```
[RPC: sanitario_recompute_agenda_core]
    ↓
[SQL: Step 1-8 de elegibilidade → Step planned → INSERT agenda_itens]
    ↓
[Deduplicação: ux_agenda_dedup_active index]
```

- **Gatilho**: `gera_agenda=true` + `ativo=true` no protocolo
- **Elegibilidade**: idade (janela_etaria), sexo, espécie, risco (raiva/carrapato/helmintos)
- **Deduplicação**: `render_sanitario_canonical_dedup_key` + index `ux_agenda_dedup_active`
- **Anti-zombie**: `unknown_history_policy='start_from_d1'` para raiva, anti-backfill expirado

### 3.3 Finalização de Agenda Sanitária

```
[RegistrarSanitarioSection]
    ↓
[trySanitaryRpcFinalize → concluirPendenciaSanitaria]
    ↓
[RPC: sanitario_complete_agenda_with_event]
    ↓
[INSERT eventos + eventos_sanitario + UPDATE agenda_itens status=concluido]
    ↓
[RPC: sanitario_recompute_agenda_core (recompute incremental)]
```

- **Idempotência**: RPC retorna evento existente se já concluído
- **Snapshot**: `sanitary_completion` gravado em `eventos_sanitario.payload`
- **Versionamento de item**: `protocol_item_version` gravado no payload (linha 1663-1665)

### 3.4 Registro Sanitário Avulso

```
[Registrar → SanitarioSection]
    ↓
[buildRegistrarActionSectionSlots]
    ↓
[resolveNonFinancialFinalizePlan → sanitaryFinalize]
    ↓
[createGesture → queue_gestures]
```

- **Payload**: `sanitary_completion` estrutura com `family_code`, `milestone_code`, `protocol_item_version_id`
- **Sem RPC direto**: Usa `createGesture` offline-first
- **Sem validação de rastreabilidade**: Não exige `produto_veterinario_id`, `lote_id`

### 3.5 Materialização de Pack Oficial

```
[OficialSanitaryPackManager → activateOfficialSanitaryPack]
    ↓
[selectOfficialSanitaryPack (filtra por aptidão/sistema/risco)]
    ↓
[buildOfficialSanitaryPackOps]
    ↓
[createGesture (offline-first)]
    ↓
[supabase.rpc("materialize_standard_sanitary_protocols")] ❌ ROMPE ATOMICIDADE
```

- **Idempotência PR4**: `reconcileFarmSanitaryFamiliesBeforePackReapply`
- **Risco**: Chamada RPC após `createGesture` sem retry/atomicidade

### 3.6 Consumo de Produto/Estoque

```
[eventoSanitario com produto_veterinario_id]
    ↓
[evaluateSanitaryInventoryConsumptionReadiness]
    ↓
[decisão: needs_catalog_product_reference | eligible_manual_consumption_source]
    ↓
[Tela /insumos → consumo manual vinculado]
```

- **Snapshot**: `custo_unitario_snapshot`, `quantidade` gravados em `insumo_movimentacoes`
- **Sem baixa automática**: Regra atual impede `consumo automatico`

## 4. Riscos P0/P1 Identificados

### P0 (Crítico)
1. **Versionamento semântico de itens** — Eventos apontam para versão estática, não evoluem com mudanças
2. **Chamada RPC após createGesture** — Quebra atomicidade offline-first, sem rollback configurado
3. **Divergência de campo `operational_complement`** — Lógica de precedência pode falhar

### P1 (Alto)
1. **Protocolos inativos não filtrados em conflitos** — `ativo=false` ignorado em famílias
2. **Edição de pack oficial sem proteção** — UI permite alterar origem oficial
3. **Desativação sem avaliação de agenda/eventos** — Não há preflight de impacto

### P2 (Médio)
1. `dedup_template` inconsistente entre coluna/payload
2. Defaults `geraAgenda/intervaloDias` indevidos
3. Normalização `family_code` com múltiplos algoritmos
4. Checklists misturados com protocolos operacionais
5. Rastreabilidade mínima não validada

- **Risco:** Médio (API pode quebrar se houver código usando `is_operational_complement`)

### Fase 2: Resolução de Protocolo Efetivo por Família
- **Duração:** 1 sprint (já implementado parcialmente)

  - Incluir filtro `ativo=true` em `findSanitaryFamilyConflict`

- **Arquivos:** `src/lib/sanitario/engine/protocolLayers.ts`
- **Risco:** Baixo (correção defensiva)
- **Testes:** `protocolLayers.test.ts`

### Fase 3: Versionamento Semântico de Itens
- **Duração:** 2 sprints
- **Entregas:**
  - UPDATE gera nova versão (`INSERT` + `ativo=false` no antigo)
  - `protocol_item_version_id` linkado a versão específica
  - Eventos preservam versão ao finalizar
- **Arquivos:** `customization.ts`, `createRegistrarFinalizeController.ts`, `sanitaryFinalize.ts`
- **Risco:** Alto (migração de dados + rollback complexo)
- **Migração:** Arquivo `supabase/migrations/202606XX_protocolo_itens_versionamento.sql`
- **Testes:** Teste de evento antigo continua apontando para versão correta

### Fase 4: Rastreabilidade do Evento Executado
- **Duração:** 1 sprint
- **Entregas:**
  - Preflight obrigatório: `produto_veterinario_id`, lote, validade, dose
  - Payload enriquecido com `validade_lote`, `dose_aplicada`, `via_aplicacao`
  - UI: validação em `RegistrarSanitarioSection`
- **Arquivos:** `src/pages/Registrar/components/RegistrarSanitarioSection.tsx`, `sanitaryFinalize.ts`
- **Risco:** Médio (UX pode ser mais restrita)
- **Testes:** Integração `evento_sem_produto → bloqueio`

### Fase 5: Separação Protocolo/Checklist/Caso Clínico
- **Duração:** 2 sprints
- **Entregas:**
  - Checklists → `dominio=conformidade` (não `sanitario`)
  - Doenças notificáveis → `sanitario_casos.tipo=notificavel`
  - Protocolos operacionais só com `gera_agenda=true` e `ativo=true`
- **Arquivos:** `src/lib/sanitario/compliance/transit.ts`, `agendaDiagnostics.ts`
- **Risco:** Médio (migração de dados)
- **Testes:** Checklist não aparece em recompute sanitário

### Fase 6: Defaults e UX Seguros
- **Duração:** 1 sprint
- **Entregas:**
  - Default `geraAgenda=false` em `createEmptyProtocolItemDraft`
  - Default `intervaloDias` vazio (não 1)
  - Proteção de edição: protocolos oficiais bloqueados
  - Normalização `family_code` unificada (Unicode-aware)
- **Arquivos:** `customization.ts`, UI de edição
- **Risco:** Baixo (defaults mais seguros)
- **Testes:** Protocolo novo sem `geraAgenda` não gera agenda
